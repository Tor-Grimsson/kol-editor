# 1.0.0 Documentation Evolution: Docs Layout Concept

## Overview

We prototyped a documentation hub inside `/styleguide/docs` to validate how markdown proofs (e.g. `0.0.0-documentation-index.md`) render inside the design-system site. The layout uses a fixed 40px outer gutter, a 3-column grid (304 / 680 / 192), and smaller type (14px base) to mirror spec-style manuals. This document captures the deviations from `kol-prose`, the component plan, and next steps for codifying the system.

## Current Progress

- Rendered the retired “Documentation System Proposal” markdown inside the new layout.
- Introduced folder tabs, breadcrumb trail, and sticky summary cards for navigation.
- Tuned typography (14px body, 48px Right Grotesk title, soft foreground tokens).
- Confirmed sticky behavior for left navigation (`StickyNavCard`) and right-hand TOC.
- Created shared docs utilities in `@kol/ui/css/docs.css` (title, meta, code blocks, layout columns).
- Extracted React wrappers (`DocsHeader`, `DocsLayout`, `DocsArticle`, etc.) for `/styleguide/docs`.
- Wired folder tags, summary cards, and collection lists to `system-metrics.json`.
- Updated the markdown parser to emit docs-specific classes for lists, meta rows, callouts, and code fences.
- Published [0.0.2 Documentation: Layout Spec](0.0.2-documentation-layout-spec.md) as the canonical reference.

## Layout Tokens

```
docs-layout        → max-width: 1320px, gutters: 40px
docs-column-nav    → width 304px, sticky stack (top: 32px)
docs-column-main   → width 680px, inner mx-10 (40px)
docs-column-toc    → width 192px, sticky TOC (top: 32px)
docs-folderbar     → horizontal pill row (uppercase mono)
```

Breadcrumb, folder bar, and article share the same horizontal alignment (40px inset on mobile, 1320px center on desktop).

## Component Checklist

1. **DocsHeader** – Inline title + subtitle + meta strip (replaces `DesPage`).
2. **DocsFolderBar** – Pill list for folder navigation.
3. **DocsBreadcrumb** – Path trail; eventually clickable.
4. **DocsSummaryColumn** – Sticky summary cards (`StickyNavCard`), filters, collections tree.
5. **DocsArticle** – Wrapper applying docs-specific type stack to markdown blocks.
6. **DocsTOC** – Sticky right column generated from headings.
7. **DocsMarkdownParser** – Maps markdown elements to docs classes (paragraph, list, code, blockquote).

## Next Steps

1. Record documentation-specific metrics (per-category chapter counts, ADR totals) in `system-metrics.json`.
2. Design patterns for tables, diagrams, and multi-column comparisons within the docs article column.
3. Explore search / filter affordances for large documentation sets (align with writing guidelines).
4. Roll the layout out to session logs, ADRs, and migration guides to validate versatility.

## Open Questions

- Do ADRs, migration logs, and research proofs share this layout or require variants?
- How should we treat tables, diagrams, and multi-column comparisons?
- What is the “print/export” strategy (PDF, copy path, open in editor)?
