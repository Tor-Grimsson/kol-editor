# 0.0.1 Metadata: Writing Guidelines

**Version:** 1.0.0
**Date:** 2025-11-03
**Status:** Active
**Content Type:** metadata

---

## Purpose

This document establishes the writing standards and conventions for all documentation in the kolkrabbi design system. Following these guidelines ensures consistency, clarity, and discoverability across 970,299 possible documents.

---

## Document Structure

### Required Frontmatter

**Every document MUST include frontmatter at the top:**

```yaml
---
version: SemVer (e.g., 1.0.0)
date: YYYY-MM-DD
status: proposal | active | deprecated | archived
content-type: research | implementation | reference | metadata
category: foundation | design-system | components | pages | content | research | operations | decisions | future
cross-references:
  parent: M.m.p  # Parent document number (if applicable)
  children:      # Array of child document numbers
    - M.m.p
    - M.m.p
  related:       # Array of related document numbers
    - M.m.p
    - M.m.p
  deprecated-by: []  # Document that replaced this (if deprecated)
  replaces: []       # Document this replaces (if new)
---
```

**Example:**
```yaml
---
version: 1.0.0
date: 2025-11-03
status: active
content-type: implementation
category: design-system
cross-references:
  parent: 2.0.0
  children:
    - 2.1.0
    - 2.2.0
  related:
    - 3.1.0
    - 6.1.0
---
```

### Title Format

**H1 must follow this format:**

```markdown
# M.m.p Category: Specific Topic Title
```

**Rules:**
- Include the M.m.p number
- Use Category (singular, capitalized)
- Use descriptive title (Title Case)
- Be specific but concise

**Examples:**
```markdown
✅ Correct:
# 2.1.0 Design System: Color System
# 6.1.0 Research: Data Table Best Practices
# 1.1.0 Foundation: Build System

❌ Incorrect:
# Color System (missing number)
# Data Table Research (missing category)
# Build tools (missing number, wrong case)
```

### Section Structure

**Use consistent heading hierarchy:**

```markdown
# H1 - Document Title (M.m.p)
## H2 - Major sections (Overview, Purpose, etc.)
### H3 - Subsections within major sections
#### H4 - Rare, only for very long documents
```

**Standard H2 sections (use as applicable):**
- **Overview** - What this document covers
- **Purpose** - Why this exists
- **Requirements** - What you need to use this
- **Implementation** - How to implement (for implementation docs)
- **Usage** - How to use (for reference docs)
- **Examples** - Code examples, screenshots
- **Related Documentation** - Cross-references
- **Status** - Current state and remaining work

## Content Guidelines

### Voice and Tone

**Voice:** Active, direct, confident
**Tone:** Professional but approachable, technical but accessible

**DO:**
- Write in active voice: "Use semantic tokens" not "Semantic tokens should be used"
- Be direct: "Do X" not "It is recommended to do X"
- Use second person: "You" not "One should"
- Use imperative mood: "Create a component" not "Creating a component"

**DON'T:**
- Use passive voice unnecessarily
- Be wishy-washy: "maybe", "perhaps", "might want to"
- Use marketing fluff or overly promotional language
- Assume knowledge - explain concepts clearly

### Language Rules

**Capitalization:**
- Design system terms: "Design System", "Atomic Design", "Semantic Tokens"
- Technology names: "React", "Vite", "Turborepo", "Sanity", "Tailwind"
- File extensions: ".jsx", ".md", ".css"
- Command names: `yarn dev`, `git commit`, `npm run build`

**Abbreviations:**
- First use: Full term + abbreviation in parentheses
  - Example: "TypeScript (TS)", "JavaScript (JS)", "Cascading Style Sheets (CSS)"
- Subsequent uses: Abbreviation only
  - Example: "TS", "JS", "CSS"

**Technical Terms:**
- Code: Use backticks: `Button`, `useState`, `--color-primary`
- Files: Use backticks: `components/Button.jsx`, `theme.css`
- Commands: Use backticks: `yarn install`, `git status`
- Properties: Use backticks: `className`, `src`, `variant`

### Code Examples

**JavaScript/JSX:**
```jsx
// Good: Include filename as comment
// components/Button.jsx
import { Button } from '@kol/ui'

export default function MyComponent() {
  return (
    <Button variant="primary">
      Click me
    </Button>
  )
}
```

**CSS:**
```css
/* Good: Include context */
.my-component {
  /* Use semantic tokens */
  color: var(--color-text-primary);
  font-size: var(--font-size-body);
}
```

**Shell/Bash:**
```bash
# Good: Include prompt for clarity
$ yarn install
$ yarn dev:web
```

### Lists

**Use consistent bullet styles:**

- Use `-` for unordered lists
- Use `1.` for ordered lists
- Nest lists with 2 spaces for first level, 4 for second
- Use sentence case for list items (capitalize first word only)

**Example:**
```
Item one
- Sub-item one
- Sub-item two
  - Sub-sub-item one
- Sub-item three
```

### Tables

**Use for structured data:**

```markdown
| Column 1 | Column 2 | Column 3 |
|----------|----------|----------|
| Value 1  | Value 2  | Value 3  |
| Value 4  | Value 5  | Value 6  |
```

**Table rules:**
- Use pipes (`|`) on both ends
- Use consistent spacing
- Left-align text (default)
- Use backticks for code in tables

**Always use this format:**

```markdown

### Cross-Reference Types

1. **Direct** - Primary related content
   - `[2.1.0 Color System](2.1.0-design-system-colors.md)`

2. **See Also** - Secondary related content

## Versioning

### Semantic Versioning

**Format:** `MAJOR.MINOR.PATCH`

**Increment rules:**

- **MAJOR** (X.0.0):
  - Breaking changes
  - Complete rewrites
  - Category changes
  - Numbering changes

- **MINOR** (1.X.0):
  - New sections added
  - Significant content additions (5+ paragraphs)
  - New sub-topics
  - New cross-references

- **PATCH** (1.0.X):
  - Typo fixes
  - Small clarifications
  - Link updates
  - Minor formatting changes

**Example:**
```yaml
version: 1.0.0  # Initial version
version: 1.0.1  # Fixed typo
version: 1.1.0  # Added new section
version: 2.0.0  # Major rewrite
```

### Status Field

**Choose ONE:**

- **proposal** - Draft, not yet approved
- **active** - Current, in use, being maintained
- **deprecated** - Outdated, replaced by another doc
- **archived** - Historical, no longer relevant

**When to change status:**
- Change to `active` when approved
- Change to `deprecated` when replaced (add `deprecated-by` to frontmatter)
- Change to `archived` when no longer relevant (after 6+ months deprecated)

## Content Type Guidelines

### Research (6.x.x)

**Purpose:** Document external best practices and findings.

**Structure:**
- Overview of research topic
- Sources consulted
- Key findings (numbered list)
- Recommendations for implementation
- Links to where findings are applied

**Rules:**
- Research is **frozen** - mark as complete, never update findings
- Use findings to inform implementation docs, don't update research
- Include all sources (URLs, books, articles)
- Date all research

### Implementation (All categories)

**Purpose:** How to build or use something.

**Structure:**
- Overview
- Prerequisites
- Step-by-step instructions
- Code examples
- Troubleshooting
- Related docs

**Rules:**
- Be practical and actionable
- Include complete examples (not just snippets)
- Test all code before documenting
- Link to research that informed decisions

### Reference (All categories)

**Purpose:** Quick lookup, specifications, API docs.

**Structure:**
- Overview
- Detailed specifications
- Examples
- Related docs

**Rules:**
- Be comprehensive
- Use tables for structured data
- Keep up-to-date with implementation
- Include version information

## File Organization

### File Names

**Format:** `M.m.p-category-topic.md`

**Rules:**
- Start with M.m.p (required)
- Use lowercase
- Use hyphens (kebab-case)
- Include category keyword
- Be descriptive but concise (3-6 words after number)

**Examples:**
```
✅ Correct:
- 2.1.0-design-system-colors.md
- 3.1.0-components-atoms.md
- 6.1.0-research-data-tables.md

❌ Incorrect:
- colors.md (missing number)
- 2.1-colors.md (missing category keyword)

# 3.1.5 Components: Button Implementation

## Overview

This document describes how to implement the Button component using the design system tokens and guidelines.

## Prerequisites

- [2.1.0 Design System: Color System](2.1.0-design-system-colors.md)
- [2.2.0 Design System: Typography](2.2.0-design-system-typography.md)

## Implementation

### 1. Import Required Tokens

```css
/* components/Button.css */
@import '@kol/ui/css/theme.css';
```

### 2. Create Button Styles

```css
.button {
  /* Use semantic tokens */
  background-color: var(--color-primary);
  color: var(--color-text-on-primary);
  padding: var(--spacing-sm) var(--spacing-md);
  border-radius: var(--radius-sm);

  /* Typography */
  font-family: var(--font-family-body);
  font-size: var(--font-size-body);
  font-weight: var(--font-weight-medium);
}
```

### 3. Use in JSX

```jsx
// components/Button.jsx
export default function Button({ children, variant = 'primary' }) {
  return (
    <button className={`button button--${variant}`}>
      {children}
    </button>
  )
}
```

# 6.1.0 Research: Data Table Best Practices

## Overview

Research on data table component design, accessibility, and best practices from industry-leading design systems.

## Sources

1. [Material Design 3 - Data Tables](https://m3.material.io/components/data-tables)
2. [Carbon Design System - Tables](https://carbondesignsystem.com/components/data-table)
3. [WCAG 2.1 - Tables](https://www.w3.org/WAI/WCAG21/Understanding/info-and-relationships.html)

## Key Findings

1. **Use semantic HTML**: Always use `<table>`, `<thead>`, `<tbody>`, `<caption>`
2. **Provide scope attributes**: Use `scope="col"` for headers
3. **Avoid fixed widths**: Use flexible layouts for accessibility
4. **Consider zebra striping**: For tables with 10+ rows
5. **Monospace for data**: Use monospace fonts for technical data

## Recommendations

- Implement responsive tables with horizontal scroll on mobile
- Use focus indicators for keyboard navigation
- Provide sort indicators for sortable columns
- Use color tokens for status states

## Implementation

These findings should inform the implementation of:
- [3.3.0 Components: Data Tables](3.3.0-components-organisms-data-table.md)
