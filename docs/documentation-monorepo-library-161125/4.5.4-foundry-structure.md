### Why Specimens are Outside `/foundry/*`

**Design Decision**: Specimen pages live at `/specimen/*` rather than `/foundry/specimen/*`.

**Foundry Pages** (`/foundry/*`) are about the **typefaces themselves**:
- Product information and features
- Technical specifications
- Licensing details
- "Here's what you can buy"

**Specimen Pages** (`/specimen/*`) are about **demonstrating typography in context**:
- Real-world applications and use cases
- Editorial layouts and design inspiration
- "Here's what you can create with it"

**Rationale**:
1. **Different Mental Models**: `/foundry/malromur` = product details, `/specimen/malromur` = design examples
2. **Different Audiences**: Foundry pages target buyers, specimens target designers seeking inspiration
3. **Scalability**: Specimens can showcase multiple typefaces together (pairings, mixed layouts)
4. **Content Type**: Foundry = product catalog, Specimen = portfolio/gallery
5. **Navigation Flow**: Specimens work as standalone showcases, shareable on social media and design platforms

This separation allows specimens to function as independent design pieces while maintaining clear connections to the typefaces they showcase.

### Specimen Organization

**Important**: Specimens are **use-case demonstrations**, not typeface pages.

- **Listing Page**: `/foundry/specimens` shows 3 typeface specimen cards
- **Hub Pages**: Each typeface has a hub page at `/specimen/{typeface}` showing all available patterns
- **Individual Specimens**: Located at `/specimen/{typeface}/{pattern}`
- **Typeface Attribution**: Each specimen clearly indicates which typeface it showcases

**URL Structure**: `/specimen/{typeface}/{pattern-name}`

## Overview

The Foundry section uses a **data-driven architecture** where all typeface pages are generated from a central configuration file. This eliminates code duplication and ensures consistency across all typeface presentations.

**Key Principle**: One typeface = one config object, not seven duplicate components.

## Usage Patterns

### 1. Route Files (Minimal Wrappers)

Route files simply fetch the config and pass it to TypefacePage:

```jsx
// apps/web/src/routes/foundry/FoundryMalromur.jsx
import TypefacePage from '../../components/sections/foundry/TypefacePage'
import { getTypefaceConfig } from '../../data/foundry/typefaceConfig'

export default function FoundryMalromur() {
  const typeface = getTypefaceConfig('malromur')
  return <TypefacePage typeface={typeface} />
}
```

**All 7 route files follow this exact pattern.**

### 2. TypefacePage Component

Receives typeface config and renders the complete page layout:

```jsx
const TypefacePage = ({ typeface }) => {
  const {
    displayName,
    fontFamily,
    fontUrl,
    fontStyle,
    badgeText,
    styles
  } = typeface

  const showVariableSection = styles.hasWeight || styles.hasWidth

  return (
    <div className="min-h-screen bg-surface-primary">
      <main className="w-full">
        <TypefaceHero typeface={typeface} />

        {/* Images */}
        <section>...</section>

        <TypefaceStyleSection typeface={typeface} />

        <FontPreviewSection
          fontFamily={fontFamily}
          badgeText={badgeText}
          showDropdown={styles.hasItalic}
        />

        {showVariableSection && (
          <VariableFontSection
            fontFamily={fontFamily}
            badgeText={badgeText}
            showDropdown={styles.hasItalic}
          />
        )}

        <GlyphMetricsGrid
          fontUrl={fontUrl}
          fontFamily={fontFamily}
          fontStyle={fontStyle}
        />

        <FoundryCharacterSets
          fontFamily={fontFamily}
          showDropdown={styles.hasItalic}
        />

        {/* Shared sections - same for all typefaces */}
        <FoundryOpentypeFeatures />
        <FoundryTypefaceDetails />
        <LicenseSection />
        <FoundryTypefacePairing />
        <FoundryOtherTypefaces />
      </main>
    </div>
  )
}
```

### 3. Parameterized Components

Components accept typeface config and adapt their behavior:

#### TypefaceHero
```jsx
const TypefaceHero = ({ typeface }) => {
  const { displayName, fontFamily, fontStyle, category, description } = typeface
  const isItalic = fontStyle === 'italic'

  return (
    <section>
      <Pill variant="subtle">{category}</Pill>
      <h1 className={isItalic ? 'italic' : ''} style={{ fontFamily }}>
        {displayName}
      </h1>
      <p style={{ fontFamily }}>{description}</p>
    </section>
  )
}
```

#### TypefaceStyleSection
Handles different typeface capabilities:

```jsx
const TypefaceStyleSection = ({ typeface }) => {
  const { fontFamily, badgeText, styles: styleConfig } = typeface
  const { hasWeight, hasWidth, hasItalic, weights, widths } = styleConfig

  // Show weight/width dropdown only for Rót
  const showDropdown = hasWeight && hasWidth

  // Determine which styles array to use
  const styles = selectedVariant === 'width' ? widths : weights

  // Build preview styles based on capabilities
  const previewStyle = {
    fontFamily,
    fontWeight: currentStyle.weight || 400,
    fontStyle: isItalic ? 'italic' : 'normal'
  }

  if (currentStyle.width) {
    previewStyle.fontVariationSettings = `'wdth' ${currentStyle.width}`
  }

  return (/* ... */)
}
```

### 4. Shared Components

Some sections are identical across all typefaces:

- `FoundryOpentypeFeatures` - Same OpenType features
- `FoundryTypefaceDetails` - Same technical details format
- `LicenseSection` - Same licensing information
- `FoundryTypefacePairing` - Same pairing suggestions
- `FoundryLicenseQuestions` - Same FAQ
- `FoundryOtherTypefaces` - Same "explore more" section

These don't accept props - they render the same content for all typefaces.

## Typeface Capabilities

The system handles 3 types of typefaces:

### 1. Variable Fonts with Weight Only
**Examples**: Málrómur, Gullhamrar, Orðspor

```javascript
styles: {
  hasWeight: true,
  hasWidth: false,
  hasItalic: true,  // Málrómur only
  weights: [
    { label: 'Thin', weight: 100 },
    { label: 'Regular', weight: 400 },
    { label: 'Bold', weight: 700 }
  ]
}
```

### 2. Variable Fonts with Weight + Width
**Example**: Rót

```javascript
styles: {
  hasWeight: true,
  hasWidth: true,
  hasItalic: false,
  defaultStyle: 'weight',
  weights: [
    { label: 'Thin', weight: 100 },
    // ...
  ],
  widths: [
    { label: 'Narrow', width: 100 },
    { label: 'Normal', width: 250 },
    { label: 'Extended', width: 400 }
  ]
}
```

Shows a dropdown to toggle between weight and width views.

### 3. Static Display Fonts
**Examples**: Dylgjur, Silfurbarki, Tröllatunga

```javascript
styles: {
  hasWeight: false,
  hasWidth: false,
  hasItalic: false,
  weights: [
    { label: 'Regular', weight: 400 }
  ]
}
```

No variable controls, single style only.

## Adding a New Typeface

To add a new typeface, only 2 steps required:

### 1. Add Configuration

Edit `apps/web/src/data/foundry/typefaceConfig.js`:

```javascript
export const typefaceConfig = {
  // ... existing typefaces

  newtypeface: {
    id: 'newtypeface',
    name: 'New Typeface',
    displayName: 'New Typeface',
    fontFamily: 'TGNewTypeface',
    fontUrl: '/fonts/TGNewTypeface-Regular.otf',
    fontStyle: 'normal',
    category: 'Display Font',
    description: 'A brief description of the new typeface',
    badgeText: 'New Typeface Aa',

    styles: {
      hasWeight: false,
      hasWidth: false,
      hasItalic: false,
      weights: [
        { label: 'Regular', weight: 400 }
      ]
    }
  }
}
```

### 2. Create Route File

Create `apps/web/src/routes/foundry/FoundryNewtypeface.jsx`:

```jsx
import TypefacePage from '../../components/sections/foundry/TypefacePage'
import { getTypefaceConfig } from '../../data/foundry/typefaceConfig'

export default function FoundryNewtypeface() {
  const typeface = getTypefaceConfig('newtypeface')
  return <TypefacePage typeface={typeface} />
}
```

**That's it!** The page will automatically render with all sections.

**Before this system**: Would have required creating 7+ component files (~170 lines)
**With this system**: ~20 lines of configuration

## Design Principles

### 1. Single Source of Truth
All typeface data lives in `typefaceConfig.js`. Changes to font paths, names, capabilities happen in one place.

### 2. Zero Code Drift
Impossible for typeface pages to have different layouts or behaviors - they all use the same components.

### 3. Configuration Over Code
Adding/modifying typefaces is a data operation, not a code operation.

### 4. Type Safety
Helper functions (`getTypefaceConfig`) provide clear APIs with fallbacks.

### 5. Conditional Rendering
Components adapt to typeface capabilities:
- Show/hide variable font sections
- Show/hide italic toggles
- Show/hide weight/width dropdowns

## Troubleshooting

### Typeface Not Showing
1. Check typeface ID in route file matches config key
2. Verify font file exists at `fontUrl` path
3. Check browser console for font loading errors

### Wrong Styles Showing
1. Verify `hasWeight`, `hasWidth`, `hasItalic` flags in config
2. Check `weights`/`widths` arrays are populated
3. Ensure `defaultStyle` is set for multi-axis fonts (Rót)

### Layout Differences Between Typefaces
This shouldn't happen! All typefaces use the same `TypefacePage` component.
If you see differences, it's a bug in the parameterization logic.

## Route Organization (2025-11-11 Update)

### Directory Restructuring

The foundry routes were reorganized to improve clarity and scalability:

**Before**:
- All 30+ files in flat `routes/foundry/` directory
- Mixed foundry pages and specimen pages
- Non-descriptive names (`SpecimenOne`, `SpecimenTwo`, etc.)
- Redundant wrapper file (`Foundry.jsx`)

**After**:
```
routes/
├── foundry/                    # Foundry (product) pages only
│   ├── typefaces/              # Individual typeface pages subfolder
│   └── [overview/licensing/specimens/typefaces pages]
│
└── specimens/                  # Specimen (showcase) pages only
    ├── gullhamrar/             # Grouped by typeface
    ├── malromur/
    └── rot/
```

1. Create `routes/specimens/{typeface}/{TypefaceFeatureName}.jsx`
2. Add route in App.jsx with descriptive component name
3. File location indicates typeface and purpose immediately

Example:
```bash
# New Málrómur specimen for book design
routes/specimens/malromur/MalromurBookDesign.jsx
```

## Key Takeaways

1. **One config object** = one complete typeface page
2. **Three parameterized components** handle all seven typefaces
3. **Adding new typeface** = 20 lines of config vs 170+ lines of components
4. **Zero code drift** - impossible for pages to diverge
5. **Type-safe** with helper functions and clear prop interfaces
6. **Organized routes** = foundry (product) vs specimens (showcase) clearly separated
7. **Descriptive naming** = instant understanding of file purpose

**The foundry structure eliminates duplication while maintaining flexibility for different typeface capabilities, and the route organization ensures scalability and developer experience.**
